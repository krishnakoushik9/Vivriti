## SECTION 1 — System Architecture

### Overall Architecture
The system implements a **microservices-based hybrid credit decisioning engine** with four primary components:

1. **Java Spring Boot Backend** (`core-backend-java:8090`) - Core orchestration and deterministic policy engine
2. **Python ML Worker** (`ml-worker-python:8001`) - AI/ML models and document processing
3. **Node.js BFF** (`bff-node:3001`) - Real-time communication and web scraping
4. **Next.js Frontend** (`frontend-nextjs:3000`) - React-based dashboard

### Architecture Diagram Description
```
Frontend (Next.js) → BFF (Node.js) → Java Backend → Python ML Worker
                      ↓                    ↓              ↓
                 Socket.io          Orchestration    4 ML Models:
                 Real-time         Service          • Random Forest
                 Progress          + Pricing        • Isolation Forest
                                   Matrix           • NLP Sentiment
                                                    • Gemini CAM
```

### Backend Framework
- **Java 21** with Spring Boot 3.2.3
- **H2 in-memory database** with JPA entities
- **JWT-based service authentication** between microservices
- **WebFlux** for async HTTP calls to Python worker
- **Prometheus metrics** and comprehensive audit logging

### AI Model Usage
- **Random Forest Classifier**: Credit risk scoring on mock CMIE Prowess data
- **Isolation Forest**: Anomaly detection for circular trading patterns
- **NLP Sentiment Analyzer**: Keyword-based financial sentiment analysis
- **Google Gemini 2.5 Flash**: CAM document generation

### Database
- **H2 in-memory** with `LoanApplication`, `RiskSignal`, `AuditLog`, `QualitativeNote` entities
- **JSON fields** for storing ML explanations and research data
- **Deterministic policy rules** stored in `PricingMatrixService`

### APIs
- **REST APIs** for application CRUD operations
- **WebSocket** via Socket.io for real-time progress updates
- **Internal service-to-service** APIs with JWT authentication

### Frontend Flow
- **Dashboard** with application list and detailed views
- **Real-time progress tracking** during ML pipeline execution
- **Tabbed interface** for financials, intelligence, explainability, and CAM
- **Socket integration** for live pipeline status updates

## SECTION 4 — Weak Points That Could Lose the Hackathon

### Architecture Weaknesses
1. **H2 In-Memory Database**: No persistence, loses all data on restart
2. **Mock Data Dependencies**: Heavy reliance on generated datasets
3. **No Real External APIs**: All Indian data sources are mocked

### ML Limitations
1. **Synthetic Training Data**: Random Forest trained on generated data, not real Indian corporate data
2. **Basic NLP**: Keyword-based sentiment instead of proper financial NLP
3. **No Model Validation**: No backtesting or performance metrics on real data

### Lack of Explainability
1. **Technical SHAP Only**: Explanations not business-friendly
2. **No Feature Importance**: Business users can't understand key drivers
3. **Black Box LLM**: Gemini CAM generation isn't explainable

### Unrealistic Assumptions
1. **Perfect Data Quality**: Assumes clean financial inputs
2. **No Data Validation**: Limited checks on extracted data quality
3. **Instant Processing**: Real-world document processing takes time

### Scalability Issues
1. **Synchronous Processing**: Pipeline blocks on slow ML models
2. **No Caching**: Repeated expensive operations
3. **Memory Constraints**: Large document processing in memory

### Demo Risks
1. **API Key Dependencies**: Gemini API required for CAM generation
2. **OCR Dependencies**: Tesseract binary may not be available
3. **Network Dependencies**: External API calls may fail during demo

```python
import camelot
def extract_financial_tables(pdf_path):
```

```python
# New file: gst_analyzer.py
def analyze_gst_compliance(gst_data):
    compliance_score = 0
    flags = []
    
    # Check filing regularity
    if gst_data.get('late_filing_count', 0) > 3:
        flags.append("Multiple late GST filings")
        compliance_score -= 20
    
    # Check ITC consistency
    if gst_data.get('itc_variance', 0) > 0.15:
        flags.append("High ITC variance detected")
        compliance_score -= 15
    
    return max(0, min(100, compliance_score)), flags
```
