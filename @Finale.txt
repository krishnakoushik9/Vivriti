* Node BFF Layer: Exposes clean REST endpoints to the UI, aggregates parallel calls between Java and Python, handles file uploads (`multer`), and enforces request rate-limiting.
* Java Spring Boot Layer: Central source of truth. Handles business rules, stores risk signals and applications in MySQL, and provides asynchronous orchestration via `WebClient`.
* Python AI/ML Layer: Heavy lifting for intelligence. Exposes FastAPI endpoints for ML predictions (Random Forest), document parsing (OCR/LLM), web scraping, and PDF generation.
* Communication Protocol: Strictly REST over HTTP using `axios` (Node), `WebClient` (Java), and `httpx/requests` (Python).

PHASE 4: DATA FLOW
===============================================================================

1. How does a PDF/annual report enter the system?
   * Endpoint: `POST /api/applications/:id/upload-document` in BFF -> routes to Python FastAPI.
   * Parser: `document_ai.py` (uses PyPDF2 + Gemini API) or `ocr_llm.py` (uses local Qwen2-VL-2B).
   * Storage: Python sends extracted structured JSON to Java `POST /api/v1/intelligence/ingest`, which saves it in MySQL `RiskSignal` table.

2. How does GST / structured data flow?
   * Ingestion: Uploaded via same document endpoint or aggregated via `aggregator.py`.
   * Usage: Pushed into the scoring pipeline (`analyze`) where extracted financial variables (revenue, EBITDA) augment the ML features.

3. How does web research / news crawling happen?
   * Agent: `research_agent.py` orchestrated by Python FastAPI.
   * APIs/Crawlers: Uses `scraper.py` which triggers DuckDuckGo HTML scraping, Google News RSS, LiveMint/ET RSS feeds, and Wikipedia API. Synthesized by Gemini API into a narrative.

4. How does qualitative input (Credit Officer notes) enter?
   * Endpoint: `POST /api/v1/applications/{applicationId}/qualitative-notes` on Java Spring Boot.
   * Usage: Stored in `QualitativeNote` table and fetched by Python during CAM generation to append human sentiment to the ML risk score.

5. How is the final CAM document generated?
   * Service: Python FastAPI `POST /api/v1/applications/{id}/export-cam`.
   * Engine: `cam_pdf_generator.py` uses ReportLab to output a fully styled PDF containing the 5 Cs, SHAP values, and extracted intelligence.

6. How does data flow from Java -> Python -> back to Java/Node?
   * REST calls over HTTP.
   * Node calls Python: `axios.post('http://localhost:8001/analyze')`
   * Python calls Java: `httpx.post('http://localhost:8090/api/v1/intelligence/ingest')`
   * Configured via `.env` and `application.yml`.

PHASE 5: ML MODELS — DEEP SCAN
===============================================================================

MODEL NAME: Random Forest Credit Risk Scorer
-------------------------
* File location: `ml-worker-python/models/rf_credit_model.pkl` (with `rf_scaler.pkl`)
* Model type: Scikit-learn RandomForestClassifier via Pipeline
* Training data: Lending Club dataset (`accepted_2007_to_2018Q4.csv.gz`)
* Input features: loan_amnt, annual_inc, dti, delinq_2yrs, revol_util, int_rate, installment, grade
* Output: Probability of default (Score 0-100)
* How is it served: Python FastAPI (`/analyze` endpoint)
* Which Java/Node service calls it: Node BFF `/api/applications/:id/analyze`
* Explainability: SHAP TreeExplainer (`explainability.py`). Returns exact feature contributions and narratives.
* Five Cs coverage: Capacity (DTI, Income), Character (Delinquencies), Conditions (Int Rate)

MODEL NAME: Vision-LLM (Qwen2-VL)
-------------------------
* File location: `ML-Model/Vision-LLM/`
* Model type: Qwen2-VL-2B-Instruct (Transformers AutoModelForImageTextToText)
* Training data: Pre-trained foundational Vision-Language Model
* Input features: Raw image tensors (scanned PDFs) + Extraction prompt
* Output: Structured JSON of financial fields (Revenue, EBITDA, etc.)
* How is it served: Python FastAPI (`/analyze-ocr-llm` endpoint)
* Which Java/Node service calls it: Node BFF fallback pipeline
* Explainability: None (Generative extraction)
* Five Cs coverage: Capital, Capacity

MODEL NAME: Gemini Pro Integration
-------------------------
* File location: API calls from `document_ai.py` & `research_agent.py`
* Model type: Gemini API (`generativelanguage.googleapis.com`)
* Training data: External
* Input features: Extracted PyPDF text, raw scraped HTML/RSS text
* Output: JSON summaries, Risk Flags
* How is it served: External REST API
* Explainability: Prompt-engineered citations (maps news links to claims)
* Five Cs coverage: Character (News, Litigation), Conditions (Sector trends)

PHASE 8: SECURITY ARCHITECTURE
===============================================================================

1. JWT TOKENS
   * Node BFF generates internal JWTs for Python using `jsonwebtoken`.
   * Python validates JWTs via Zero-Trust middleware in `main.py` using `jose` and HS512 with derived key.
   * Secret: `VivritiIntelliCreditSecretKey2025AES256BitKeyForProduction`.

2. AUTHORIZATION
   * Spring Security is imported (`SecurityConfig.java`) but largely disabled/permissive (`/**` allowed) with a note: "Disable default security for MVP demo". Basic Auth exists for an `admin` user.
   * Python strictly enforces the `validate_jwt` dependency on internal routes.

3. API SECURITY
   * Node BFF implements `express-rate-limit` (100 req/15min) and `helmet` for HTTP headers.
   * CORS is explicitly configured in Java (`application.yml` allowed-origins: 3000, 3001) and Node BFF (`cors` middleware).

4. DATA SECURITY
   * `AesEncryptionUtil.java` is present in Java, indicating potential at-rest encryption capabilities, though DB mostly stores plaintext JSON currently.

5. OVERALL SECURITY GRADE: B
   * Justification: Great Zero-Trust service-to-service implementation via internal JWTs. Rate limiting and CORS are well-configured. Docked points for disabled frontend-to-BFF authentication and hardcoded secrets in code (MVP setup).
